Flujos de trabajo de GIT

En este post te enseñaré el flujo de trabajo de GIT, un sistema de control de versiones que ayuda a llevar un registro de los cambios en tu código y colaborar con otros desarrolladores.

Comandos básicos de GIT

Estos son los comandos fundamentales que se utilizan en el día a día con Git:

Comando Descripción
git init Inicializa un nuevo repositorio
git clone Crea una copia local de un repositorio remoto
git add Añade archivos al área de preparación
git commit Guarda los cambios en el repositorio local
git push Envía los cambios al repositorio remoto
git pull Obtiene cambios del repositorio remoto
git checkout Cambia entre ramas o restaura archivos

Flujo de trabajo básico

  1. Clonar el repositorio: git clone https://github.com/usuario/proyecto.git
  2. Crear una rama: git checkout -b nueva-funcionalidad
  3. Añadir cambios: git add . (añade todos los archivos modificados)
  4. Confirmar cambios: git commit -m "Descripción de los cambios"
  5. Subir cambios: git push origin nueva-funcionalidad

Git Add explicado

El comando git add mueve los archivos modificados al área de preparación (staging). Es como una "sala de espera" antes de confirmar definitivamente los cambios.

Ejemplos de uso:

  • git add archivo.html - Añade un archivo específico
  • git add *.css - Añade todos los archivos CSS modificados
  • git add . - Añade todos los archivos modificados

Git Checkout explicado

Este comando es muy versátil y permite:

  • Cambiar entre ramas: git checkout nombre-rama
  • Crear y cambiar a una nueva rama: git checkout -b nueva-rama
  • Descartar cambios en un archivo: git checkout -- archivo.txt

Pull Requests

Los Pull Requests son una característica de plataformas como GitHub que permiten revisar código antes de fusionarlo:

  1. Hacer push de tu rama: git push origin tu-rama
  2. En GitHub, ir a "Pull requests" → "New pull request"
  3. Seleccionar la rama base (main) y la rama con tus cambios
  4. Añadir título y descripción
  5. Crear el pull request

Después de la revisión, el código puede ser fusionado a la rama principal.

Flujos de trabajo comunes

Gitflow

Un modelo de ramificación con ramas específicas:

  • master/main: Código de producción estable
  • develop: Rama de integración
  • feature/*: Nuevas funcionalidades
  • release/*: Preparación para release
  • hotfix/*: Correcciones rápidas

GitHub Flow

Un flujo más simple:

  1. Crear una rama desde main
  2. Hacer cambios y commits
  3. Abrir un Pull Request
  4. Revisar y discutir
  5. Fusionar a main
